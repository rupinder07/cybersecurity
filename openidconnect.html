OpenId Connect

OpenID Connect (OIDC) is an authentication protocol and identity layer built on top of OAuth 2.0 , designed to provide secure and standardized user authentication and identity verification on the internet. It extends OAuth 2.0's capabilities by adding an authentication layer, making it a powerful tool for single sign-on (SSO) and user identity management. OIDC is commonly used in scenarios where web applications or services need to verify the identity of end-users and obtain information about them.

Key components of OpenID Connect

  1. Relying Party (RP) The Relying Party is a web application or service that relies on OIDC for user authentication and identity verification. The RP initiates the authentication process and receives identity information from the Identity Provider (IdP)
  2. Identity Provider (IdP) The Identity Provider is responsible for authenticating users and providing identity information to the RP. IdPs are trusted entities that confirm the user's identity using various authentication methods, such as passwords, multi-factor authentication (MFA), or social logins.

How OpenID Connect Works

  1. RP Initialization

    The user initiates the authentication process by accessing a protected resource on the RP, which requires authentication. The RP generates a unique request identifier called nonce.

  2. Authentication Request

    The RP constructs an authentication request, including parameters like the client ID, scope (defining the requested identity claims), nonce, and a redirect URI where the user will be sent after authentication.

  3. User Redirection

    The user is presented with an authentication prompt at the IdP. The user provides their credentials (e.g., username and password) or uses another authentication method (e.g., biometrics).

  4. Authorization and Consent

    After successful authentication, the IdP may present a consent screen to the user, showing the requested identity claims. The user consents to sharing this information with the RP.

  5. ID Token Issuance

    Upon user consent, the IdP issues an ID token, which is a JWT containing user information (claims) and is signed by the IdP. The ID token typically includes the user's name, email, subject identifier, and any additional claims requested by the RP.

  6. Token Response

    The ID token is sent back to the RP via the user's browser and the redirect URI.

  7. RP Verification

    The RP verifies the ID token's signature, checks the nonce to prevent replay attacks, and ensures the audience claim matches its own client ID. Additional security checks may include verifying the token's expiration and issuer.

  8. User Session

    The user is considered authenticated, and their session is established with the RP. The RP may provide access to protected resources or personalize the user experience based on the received identity information
